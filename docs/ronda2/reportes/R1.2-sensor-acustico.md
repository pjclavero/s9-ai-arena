# Reporte R1.2 — Reparar el sensor acústico muerto y su test vacuo

- **Tarea:** R1.2 (banda R-P0) · **Error:** ERR-ENG-01
- **Severidad:** Crítica (funcional) · **Equipo:** E2
- **Commit:** `<pendiente>` · **Estado:** ✅ Hecho y verificado
- **Fecha:** 2026-07-17

## Problema

El sensor acústico nunca percibía nada. En `apps/arena-engine/src/sim/battle.ts`, el tick de
decisión hacía `this.sounds = []` **justo antes** del bucle que construye las observaciones, así que
los sonidos acumulados durante los 3 ticks del ciclo (disparos, impactos, minas, motores) se
borraban antes de que ninguna observación los leyera. `sensors.ts` filtraba sobre un array vacío y
devolvía siempre `sources: []`.

Agravantes:
- `observationFor()` leía `this.sounds` en otro instante → las dos rutas de observación veían
  conjuntos distintos.
- El test que debía cogerlo (`sensors-fog.test.ts`) estaba guardado con
  `if (acoustic && acoustic.sources.length > 0)` y usaba bots que no generaban sonido: **pasaba
  doblemente en vacío**.

## Solución aplicada

**Doble búfer** en `battle.ts` (solo este fichero + su test):

- Nuevo campo `observedSounds`: conserva el búfer de sonidos del ciclo de decisión **anterior**, ya
  congelado. Es lo que oyen las observaciones de este tick.
- `sounds` sigue siendo el **acumulador** del ciclo en curso (física y combate empujan ahí, sin
  cambios).
- En el tick de decisión, `this.observedSounds = this.sounds` **antes** del bucle de observaciones,
  y el vaciado `this.sounds = []` se mueve a **después** de construirlas (nunca antes — ese era el
  bug). Como el vaciado reasigna un array nuevo (no muta), `observedSounds` conserva el ciclo
  anterior.
- **Unificación:** `observationFor()` pasa a leer `this.observedSounds`, el mismo búfer que sirve el
  bucle de decisión. Ambas rutas ven exactamente el mismo conjunto para el mismo tick.

Efecto: un disparo en el tick T se percibe en la decisión del ciclo siguiente — el bot oye lo que
sonó durante el ciclo que acaba de terminar.

## Verificación

- **Test endurecido (reproducción):** monta un oyente con `sensor.acoustic` (alcance 60 m) y un
  artillero que dispara hacia el oeste (el proyectil no hiere a nadie; solo el fogonazo entra en
  alcance). Comprueba lo que **recibe el bot en `decide()`** (la ruta real que quedaba muda), no
  `observationFor()`. Exige `gunshot` con `bearing ≈ ±π` (dirección aproximada) y **sin**
  `position`/`distanceM`/`entityId`. Falló antes del fix (`expected 0 to be greater than 0`); pasa
  después.
- **Golden del motor intactos** (`git diff tests/golden/` vacío): el sonido no entra en
  `stateHash()` ni consume RNG, así que ni el hash ni la aleatoriedad cambian. `replay-golden`
  (round-trip que compara todos los hashes intermedios) sigue verde. **Determinismo preservado.**
- **Suite motor + catálogo:** `npx vitest run packages/module-catalog apps/arena-engine` →
  **190 tests verdes**. `npm run lint` (determinismo) OK.

## Notas

- El bug solo mataba la **ruta de decisión**, no `observationFor()`: con el código buggy, un test que
  llamara a `observationFor()` justo tras un `step()` con disparo habría visto el sonido y habría
  pasado en falso. Por eso el test endurecido comprueba la observación que recibe el bot en su
  `decide()`. Esto ilustra por qué unificar las dos rutas era parte del arreglo.
- Con esto, el sensor acústico ya funciona a nivel de motor. Falta **R1.3** (publicar
  `sensor.acoustic` y `sensor.proximity` como datos del catálogo) para que un vehículo real pueda
  montarlo desde un loadout.
