# R6.1 · Construir los runtimes, fijar digests reales y probar el sandbox de verdad

**Fecha:** 2026-07-17 · **Dónde:** VM108 `s9-arena` (Docker real) + GitHub Actions
**Commits:** `c1e9a2b`, `c42df40`, este · **Rama:** `ronda2/entrypoints-servicios`

## Resumen

**La suite de escape del sandbox se ha ejecutado VIVA y en verde por primera vez: 7/7
vectores contenidos con prueba**, leyendo la imagen de `runtimes/DIGESTS.lock` y
descargándola de GHCR. Hasta hoy esa garantía estaba *declarada* pero nunca *probada*:
R1.5 y R1.6 dejaron el pipeline rechazando por diseño (fail-closed) y el CI saltando los
jobs de sandbox, a la espera de esto.

R6.1 no era "rellenar hashes". De los siete problemas que hubo que resolver, **cuatro eran
defectos reales de seguridad o de arranque** que solo podían aparecer al construir y
ejecutar de verdad.

## Lo que se encontró (y por qué importaba)

### 1. Dependency confusion en los dos runtimes ← el más grave

`runtimes/python/allowed-requirements.lock` pedía `arena-sdk==1.0.0` **a PyPI**, y
`runtimes/node/allowed-package.json` pedía `@arena/sdk` **a npmjs**. Los dos son SDK
propios del repo (`sdks/python`, `sdks/javascript`), y **ni ese nombre ni ese scope son
nuestros en los registros públicos**: cualquiera podía publicar un `arena-sdk` en PyPI y su
código habría entrado en el runtime donde se ejecutan los bots. El Dockerfile ni siquiera
copiaba el SDK: solo hacía `pip install` del lock.

**Corregido:** ambos SDK se construyen desde el repo en una etapa `sdk-builder` y se
instalan desde el árbol local (`--no-index` en Python; copia directa en Node), sin tocar
ningún registro. El wheel de Python **es reproducible byte a byte** (mismo sha256 en builds
repetidos: el toolchain lo aporta la base fijada por digest y se construye sin red con
`SOURCE_DATE_EPOCH` fijo), así que su hash se fija en `runtimes/python/sdk-wheel.lock` y el
build **falla** si el SDK cambia sin actualizarlo. Hay tests de regresión para las dos vías.

### 2. El sandbox no arrancaba: 0/7 vectores NO PROBADO

Primera ejecución con la imagen real: **los 7 vectores `NO PROBADO`, exit 127**. No era un
escape — los contenedores morían antes de ejecutar una línea. Causa: el perfil
`seccomp-bot.json` es una allowlist con `defaultAction: ERRNO` que **no permite las
syscalls que `runc init` necesita para arrancar** con Docker 29 / runc moderno.

Conjunto mínimo verificado **por ablación** (quitar cualquiera de las 6 rompe el arranque):
`fstatfs`, `capget`, `capset`, `setuid`, `setgid`, `setgroups`.

Lo importante es lo que **no** hizo falta: **ninguna syscall de la API de montaje**
(`fsopen`/`fsmount`/`fsconfig`/`move_mount`/`open_tree`), aunque el mensaje de error las
señalaba. `mount`, `umount`, `unshare`, `setns`, `pivot_root` y `ptrace` **siguen
denegadas**. `setuid`/`setgid`/`setgroups` las usa runc para bajar a `botuser` (uid 10001)
antes del `exec`; con `--cap-drop ALL` y sin `CAP_SETUID`/`CAP_SETGID` un bot no puede
escalar con ellas.

Este fallo estaba **latente**: cualquiera que hubiera lanzado un bot real en VM108 se
habría encontrado con que el sandbox no arranca.

### 3. La suite no ejecutaba el bot

`run-escape-suite.sh` lanzaba `docker run ... "$IMAGE" python /bot.py` sobre una imagen con
`ENTRYPOINT ["python"]` y `WORKDIR /bot` → el comando efectivo era `python python /bot.py`
→ `can't open file '/bot/python'`. Ahora monta el bot en `/bot/main.py` y **no pasa
comando**, que es exactamente lo que hace `DockerContainerRunner.buildRunArgs` en
producción: la suite prueba el mismo camino que el pipeline real.

### 4. El vector 06 exigía un imposible (hallazgo de la SUITE, no del sandbox)

Con todo lo anterior arreglado: **6/7, y el vector 06 emitía `LEAK /proc/1/environ`**.

Se investigó antes de tocar nada, y **no hay escape**. Dentro del contenedor **PID 1 es el
propio bot** (el contenedor tiene su namespace de PIDs): solo se ve un PID, el `/sbin/init`
del host no es visible, y el `environ` leído es el suyo propio — se confirmó con un canario
en `--env`. Un proceso que es PID 1 **siempre** puede leer su propio environ: exigir que
falle no es una garantía de seguridad, es una condición imposible, y el vector no habría
podido quedar contenido nunca.

**Decisión (consultada y aprobada):** se corrige la aserción para probar el invariante
**real y comprobable**, sin relajar nada:

- el namespace de PIDs está aislado (si `/proc/1` fuese el init del **host**, ahora sale
  como `ESCAPE-CRITICAL`; antes ni se miraba);
- no hay procesos ajenos visibles en `/proc`;
- el env del bot no contiene secretos (el contrato de `buildRunArgs` dice "env NUNCA
  secretos"): si apareciera un `TOKEN`/`SECRET`/`JWT`, eso **sí** sería `LEAK`;
- `/proc/kcore`, `/run/secrets`, el token de service account y `docker.sock` siguen fuera
  de alcance.

El vector prueba ahora **más** que antes, no menos.

### 5. Tres mentiras del MVP en la allowlist

- **Versión `1.0.0` → `0.1.0`.** Es la que declaran los dos SDK (`pyproject.toml` y
  `package.json`). No existía ningún artefacto con la versión `1.0.0`.
- **`websockets` → `websocket-client`.** Son **paquetes distintos**. El SDK hace
  `import websocket` y su `pyproject.toml` depende de `websocket-client>=1.7,<2`.
  `websockets` no lo usaba nadie: el runtime instalaba una cosa y el pipeline permitía otra.
  Como el módulo no se llama como el paquete, la traducción vive ahora en
  `static-analysis.ts` (`PYTHON_IMPORT_TO_DIST`), y no metiendo `websocket` en la allowlist
  — eso habría hecho que la allowlist dejara de nombrar paquetes instalables.
- **El `integrity` de `ws` era una cadena de ceros**, así que `npm ci` **no verificaba
  nada**. Ahora es el real del registro.

### 6. El SDK de JavaScript nunca se había compilado

`sdks/javascript` es TypeScript con `noEmit: true`, `private: true` y `main: src/index.ts`.
Como paquete **no existía**: los bots de ejemplo lo importaban por ruta relativa del
monorepo. El runtime lo emite ahora a JS + `.d.ts` con `tsc` y reescribe su `package.json`
apuntando a `dist/index.js`. Verificado: un bot hace `import * as sdk from "@arena/sdk"` y
`import WS from "ws"` **de verdad** dentro de la imagen.

Los paquetes van a `/node_modules`, **no** a `/opt/runtime` con `NODE_PATH`: `NODE_PATH`
solo lo respeta CommonJS y el SDK es ESM, así que con `NODE_PATH` el import fallaba con
`ERR_MODULE_NOT_FOUND`. La resolución ESM sube por los `node_modules` de los directorios
padre, y desde `/bot` encuentra `/node_modules`.

### 7. El CI del sandbox no podía funcionar ni con digests reales

El job `sandbox-escape` de `e6-security.yml` hacía:

```yaml
run: docker build -t "${{ needs.digests-gate.outputs.python_image }}" runtimes/python
```

Eso **no puede funcionar**, y estaba oculto porque el `digests-gate` saltaba el job:

1. `-t` **no admite una referencia por digest**. El digest lo asigna el registry al
   publicar; no se le puede imponer a un build local. El paso fallaba en seco.
2. El contexto era `runtimes/python`, pero el Dockerfile hace `COPY sdks/python`: el
   contexto tiene que ser la **raíz** del repo.
3. Aunque funcionase, **reconstruir contradice el objetivo**: hay que probar la imagen
   **exacta** que está fijada, no una recién hecha.

Ahora **descarga por digest** y falla si no existe (fail-closed, como exigen R1.5/R1.6).

### 8. Los runtimes arrastraban 8 CVE críticas — y el escáner nunca se había ejecutado

Al abrir el `digests-gate`, el job `runtime-vuln-scan` corre **por primera vez** y aparecen
dos cosas:

**a) El job no podía funcionar.** Fallaba con `trivy: command not found` (exit 127): el
workflow **nunca instalaba Trivy**. Y el script decía en su comentario que "construye las
imágenes desde los digests fijados", pero hacía `docker build`: **no usaba los digests en
absoluto**. Escaneaba una imagen recién hecha, no la pineada — y como las imágenes no son
reproducibles bit a bit, podían no ser la misma: el escaneo no garantizaba nada sobre lo
que se ejecuta. Ahora descarga por digest y escanea **esa**, y falla cerrado si Trivy no
está o la imagen no existe.

**b) Con Trivy funcionando, el hallazgo real: 8 CVE CRÍTICAS**, todas con fix publicado.

| Runtime | Críticas | Paquetes |
|---|---|---|
| python | 6 | `libexpat1` (CVE-2024-45491, CVE-2024-45492), `libgnutls30`, `libssl3` |
| node | 2 | `libgnutls30` (CVE-2026-33845, CVE-2026-42010) |

Causa: las bases fijadas eran viejas — `python:3.11.9-slim-bookworm` es de 2024.

**Corregido (decisión consultada):** se sube la base a `python:3.11-slim-bookworm` (hoy
3.11.15) y `node:20-bookworm-slim`, **manteniendo el contrato de lenguaje** (Python 3.11 /
Node 20): un bot no depende del número de parche. Medido: la base nueva de python sale
limpia, pero **la de node no** — Debian 12 aún no recoge en la imagen el `gnutls` corregido
que ya tiene en sus repos —, así que ambos Dockerfile aplican además los parches del SO.
**Resultado verificado con Trivy: las dos imágenes quedan en 0 críticas.**

Efecto secundario que confirma el diseño: subir la base cambió el hash del wheel del SDK
(`9025098809…` → `55c041a46d…`) y **el build falló** hasta actualizar `sdk-wheel.lock`. El
guard hace exactamente lo que debe.

### 9. El verificador no ataba el `FROM` a la base del lock

`verify-runtime-digests.ts` comprobaba que el `FROM` llevara un `@sha256`, pero **no que
fuera el mismo que declara `DIGESTS.lock`**. Se podía subir la base y olvidar el lock (o al
revés) sin que nada protestase, dejando el lock describiendo una imagen que nadie
construye. Ahora se comparan — y se comprobó que el check **falla ante esa deriva** antes
de corregirla, en vez de darlo por bueno.

## Decisión de diseño: de dónde sale el digest (opción (a), GHCR)

Un `docker build` local no produce digest. Se publica en GHCR con
`.github/workflows/runtimes-publish.yml`, **sin ningún secret**: `GITHUB_TOKEN` ya escribe
en `ghcr.io/<owner>/*` con `permissions: packages: write`. **Ninguna credencial toca
VM108**, que descarga de forma anónima (el repo es público). Verificado.

### Hallazgo que cambió el diseño: las imágenes NO son reproducibles

Se midió, no se supuso. El **mismo commit**, publicado **dos veces por el mismo workflow**
en el mismo tipo de runner, dio digests **distintos**:

| Publicación | `bot-runtime-python` |
|---|---|
| 1ª | `sha256:4795fa54…` |
| 2ª (re-run, mismo SHA) | `sha256:2ffb8119…` |

(Las capas llevan mtimes y metadatos del momento del build.) Consecuencia práctica:
**reconstruir una imagen no sirve para verificar nada** — un digest distinto no prueba que
el runtime haya cambiado, ni uno igual que no.

Por eso el modelo final es **construir una vez → fijar → verificar descargando**, y por eso
el workflow se lanza **a mano** (`workflow_dispatch`) y no en cada push: publicar en cada
push dejaría `DIGESTS.lock` obsoleto al instante, fijando un digest que ya nadie prueba.

**Limitación conocida que esto deja abierta** (documentada en `DIGESTS.lock`): nada ata
criptográficamente el fuente de `runtimes/` a la imagen fijada. Si alguien cambia un
runtime y no republica, el CI seguirá probando la imagen anterior sin avisar. Cerrarlo pide
un build reproducible (`SOURCE_DATE_EPOCH` + `rewrite-timestamp` de buildkit) y es trabajo
aparte de R6.1.

## Tests de regresión añadidos

En `apps/bot-manager/tests/runtimes.test.ts`, para que nada de esto vuelva:

- el SDK propio **nunca** se resuelve desde PyPI (el lock no puede nombrar `arena-sdk`; el
  Dockerfile debe copiar `sdks/python` y usar `--no-index`);
- el SDK Node **nunca** se resuelve desde npmjs;
- **ningún hash placeholder** en los locks (ni `sha256:000…`, ni el `integrity` de ceros);
- la allowlist nombra `websocket-client` y **no** `websockets`;
- un bot que hace `import websocket` queda permitido pese a que el paquete se llame distinto.

En `digest-guard.test.ts`, el test que afirmaba *"el repo lleva placeholders"* se **invierte**
al estado real (`placeholderViolations() === []`), como su propio comentario anticipaba. Que
el guard los sigue detectando si vuelven lo prueban los tests que se los pasan a mano.
