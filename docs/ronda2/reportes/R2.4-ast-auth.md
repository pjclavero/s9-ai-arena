## Frente E6 · bot-manager: análisis del AST real (ERR-SEC-06)

El análisis estático parseaba los bots con **regex por línea**: no veía un `import socket`
sangrado dentro de un `if`, ni `__import__("o"+"s")`, ni `require(variable)`, ni `eval`.
Ahora se parsea el **árbol de verdad** de cada runtime:

- **Python:** el módulo `ast` de CPython (`python3 -I -c` con un analizador embebido,
  un solo proceso por análisis, todos los ficheros por stdin/JSON). Es el mismo
  intérprete que ejecuta los bots en su runtime.
- **JS:** `acorn` (parser ECMAScript real, nueva devDependency raíz). ESM primero,
  CJS como respaldo; los `.ts` en bots Node se **rechazan** (el runtime no ejecuta
  TypeScript: analizar "lo que parezca JS" sería analizar otro árbol).

Detecciones nuevas (bloquean **siempre**, con cualquier política, porque derrotan al
análisis): `__import__`, `importlib.import_module` (también vía alias, por atributo),
`eval`/`exec`/`compile` — incluso solo **referenciados** (`f = eval`), acceso a
`__builtins__`/`__loader__`/`__spec__`, `import()`/`require()` con argumento no literal
y el constructor `Function`. `re.compile` sigue siendo legítimo (el atributo `compile`
no se banea, solo el nombre desnudo).

**Fail-closed:** si un fichero no parsea (sintaxis inválida, `python3` ausente, salida
corrupta, timeout), el bot se **rechaza** con el fichero y el motivo señalados. Nunca se
aprueba lo que no se ha podido analizar.

Listas endurecidas (`apps/bot-manager/src/config.ts`):

- `os` sale de la stdlib inocua de Python y `process` de los builtins permitidos de Node.
- Python peligrosos: + `os, pty, signal, resource, importlib, runpy, code, codeop,
  builtins, compileall, py_compile, pickle, marshal, shelve, dill, shutil, pathlib,
  tempfile, glob` (antes solo `socket, subprocess, ctypes, multiprocessing, asyncio`).
- Node peligrosos: + `process, module (createRequire), v8 (serialización), repl,
  inspector, wasi, http2`.

Ficheros: `apps/bot-manager/src/ast-analysis.ts` (nuevo), `static-analysis.ts`,
`config.ts`, `pipeline.ts` (la etapa `static_analysis` falla por parseError o hallazgo
dinámico antes de cualquier otra comprobación, con hallazgo de auditoría `high`).

## Frente E7 · apps/api: endurecimiento de auth

### ERR-SEC-07 · desactivar 2FA exige reautenticación fuerte

`DELETE /auth/2fa` ya no se conforma con el access token: exige **contraseña + TOTP o
código de recuperación** (401 si faltan o son inválidos; 409 si el 2FA no está activo).
Cambiar el estado del 2FA (activar **o** desactivar) **revoca el resto de sesiones** del
usuario y audita (`auth.2fa.enabled/disabled` con `otherSessionsRevoked`). El contrato
(`openapi.yaml`) declara el requestBody y la nueva extensión `x-reauth: true`, que la
matriz RBAC generada entiende: para operaciones x-reauth, 401/409 con rol suficiente
pero sin credenciales frescas es el comportamiento **correcto**.

### ERR-SEC-08 · familias de refresh tokens + vida absoluta + rate-limit

Nueva migración `009_r24_refresh_families`:

- `session_refresh_tokens`: historial de TODOS los hashes emitidos por sesión
  (= familia); `token_hash UNIQUE`. Las sesiones vivas preexistentes se materializan
  como cabeza de familia (backfill en la migración).
- `sessions.absolute_expires_at`: tope **absoluto** (30 días, `REFRESH_ABSOLUTE_TTL_S`).
  La rotación renueva la ventana deslizante (14 días) pero **nunca** más allá del tope.

Presentar un refresh **ya rotado** = robo detectado → se revoca la **familia entera**
(la sesión y con ella todos sus tokens y el access token) y se audita
`auth.refresh.reuse_detected` con IP y motivo. El endpoint tiene rate-limit propio
(60/min por IP, inyectable en tests).

### ERR-SEC-11 · anti-enumeración en login

Cuando el email no existe, el login ejecuta **Argon2id contra un hash señuelo**
(aleatorio por proceso): ambas ramas pagan el mismo coste y el timing no delata si la
cuenta existe. Cuerpo del 401 idéntico en ambos casos.

## Omisiones declaradas (fail-closed, no aprobado)

- El análisis AST de Python depende de `python3` en el host del bot-manager: si falta,
  TODOS los bots Python se rechazan (fail-closed). En el contenedor de producción hay
  que garantizar `python3` (los runtimes lo llevan; el servicio API debe llevarlo también).
- `getattr`/reflexión más allá de los nombres baneados no es detectable estáticamente:
  sigue siendo terreno del sandbox (T6.2/R6.1), que es la defensa principal.
- No se ha probado la migración `009` contra una BD de producción con sesiones vivas
  reales (solo el backfill en embedded-postgres de los tests).
