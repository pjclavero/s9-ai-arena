# R1.6 · El CI del sandbox deja de pasar en verde sin probar (ERR-SEC-04)

**Estado:** ✅ Hecho y verificado (Linux, VM108) — con una salvedad honesta sobre la ejecución viva (ver "Lo que sigue bloqueado").

**Issue relacionado:** ERR-SEC-04 (auditoría consolidada 2026-07-16, §198).

## El fallo

El job `sandbox-escape` daba **siempre verde sin ejecutar nada**, y ese verde ocultaba ERR-SEC-03. Cuatro defectos que se reforzaban entre sí:

1. El workflow pasaba un digest literal inexistente: `run-escape-suite.sh arena/bot-runtime-python@sha256:PENDIENTE`.
2. Cada `docker run` terminaba en `|| true`, así que el fallo de arranque se tragaba.
3. Ningún bot emitía prueba positiva de contención: solo texto libre (`blocked …`). El harness únicamente buscaba marcadores de escape.
4. Resultado: contenedor que no arranca → salida sin `ESCAPE-OK` → `>>> OK: contenido` → job en verde.

O sea: **el silencio se interpretaba como contención**. Un sandbox que no existe y un sandbox perfecto producían exactamente el mismo verde.

## Qué se ha hecho

### 1. El digest placeholder se rechaza (`scripts/assert-real-digest.ts`)

Nuevo script que expone a shell el guard que ya existía (`apps/bot-manager/src/digest-guard.ts`), sin duplicar lógica. Además del placeholder de ceros que cubre el guard, exige la **forma canónica** `@sha256:<64 hex>`: `assertRealDigest` por sí solo no habría rechazado `@sha256:PENDIENTE`, que era justo lo que usaba el CI.

### 2. La imagen debe existir antes de empezar

`docker image inspect` previo. Sin imagen no hay prueba, y sin prueba la suite no puede dar verde: sale con código 2.

### 3. El código de salida es evidencia (`run-escape-suite.sh`)

Fuera el `|| true`. Se captura `rc` por separado y se distingue lo que antes se confundía:

- `125/126/127` → el contenedor **no llegó a ejecutarse** → vector **NO PROBADO** → suite en rojo.
- `137/139` → el contenedor **murió por un límite** (OOM killer, `--pids-limit`) → eso **es** la contención, y se acepta solo si el bot lo declara en `killExitCodes`.

### 4. Prueba positiva por vector (silencio = fallo)

Los 7 bots emiten ahora `SANDBOX-BLOCKED <id>` **solo tras comprobar que su ataque falló** (todos los destinos rechazados, ninguna escritura lograda, ninguna fuga leída…). El harness exige ese marcador —o un `killExitCodes` declarado— y `manifest.json` lo documenta por bot. Un bot que no se ejecuta no lo emite, y su vector se declara NO PROBADO.

### 5. El CI queda *skipped*, nunca *passed* (`.github/workflows/e6-security.yml`)

Job `digests-gate` que decide si hay con qué probar. `runtime-vuln-scan` y `sandbox-escape` llevan `if: needs.digests-gate.outputs.real_digests == 'true'`: con placeholders los jobs salen **grises (skipped)**, no verdes, y el gate deja un `::warning::` señalando R6.1. La imagen sale de `DIGESTS.lock`, no de un literal del workflow: el `@sha256:PENDIENTE` incrustado era el origen del verde falso.

## Verificación (VM108, Docker real)

**El caso que antes daba verde falso, ahora falla:**

```
$ ./tests/sandbox-escape/run-escape-suite.sh
suite de escape: digests placeholder: ejecuta el build real
  (suite de escape: arena/bot-runtime-python@sha256:1111…)
ERROR: la suite NO se ejecuta con un digest placeholder
exit=2
```

Antes, ese mismo comando recorría los 7 bots, imprimía `>>> OK: contenido` en cada uno y salía 0.

- `npx vitest run apps/bot-manager` → **80/80** (14 ficheros), incluida `sandbox-escape-suite.test.ts`, que valida la consistencia del manifiesto con los bots.
- Suite completa del repo: **702 · 0 · 3 skipped**, sin cambios.

## Lo que sigue bloqueado (y por qué no es R1.6)

La **ejecución viva** de la suite contra el sandbox real no se ha podido hacer: **el runtime de Python no es construible hoy**. Se intentó de verdad en VM108, con Docker y un registry local:

1. `runtimes/python/Dockerfile` arranca con `FROM python@sha256:0000…` — no resuelve. Se resolvió el digest real de `python:3.11.9-slim-bookworm` (`sha256:8fb09919…`) y el build avanzó.
2. Entonces falla en `pip install --require-hashes -r allowed-requirements.lock`: **el lock tiene hashes placeholder de ceros**, y el propio fichero lo admite (*"los hashes se rellenan al generar la imagen; aquí quedan los placeholders del MVP"*).

Eso es **R6.1** (construir los runtimes y fijar digests reales), no R1.6. El cambio de prueba en el Dockerfile se revirtió para no dejar el repo en un estado intermedio incoherente con `DIGESTS.lock`.

**Esto no debilita R1.6, lo confirma:** el objetivo era que la ausencia de sandbox probado dejara de fingir verde. Hoy la suite dice exactamente la verdad — "no puedo probar nada con estos digests" — y el CI se salta el job en vez de aprobarlo. Cuando R6.1 fije los digests reales, el gate se abrirá solo y la suite se ejecutará de verdad, sin tocar una línea más.
