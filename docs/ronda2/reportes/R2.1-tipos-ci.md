- **Tarea:** R2.1 (banda R-P1) · **Errores:** ERR-GES-02, ERR-GES-03 (y parte de ERR-GES-05)
- **Severidad:** Media (gestión/CI) · **Equipo:** transversal
- **Commits:** `c110bcb` (tipos) · `5781faa` (CI) · `17a2849` (formato mecánico) · **Estado:** ✅ Hecho y verificado
- **Fecha:** 2026-07-17 · **Rama:** `ronda2/r2.1-tipos-ci` (base `ronda2/entrypoints-servicios`)

## Problema

La CI decía "tipos OK" sin que fuera verdad, por dos capas:

1. **ERR-GES-02:** el `tsconfig.json` raíz incluía `apps/web` sin `jsx` → **~238 falsos errores**
   de JSX ahogaban los reales (267 errores totales en el baseline de esta rama).
2. **ERR-GES-03:** quedaban **29 errores genuinos** (la auditoría contó 38; `pipeline.ts` y parte
   de bots/battles ya se habían corregido en R1.4–R1.8, y apareció 1 nuevo en
   `battle-stats-canonical.test.ts` al exigir `BotAgent.botId`). Nadie los veía porque el paso de
   tipos llevaba `continue-on-error: true` (`ci.yml`). Además el paso "Formato" era un `echo` y no
   había cobertura (ERR-GES-05).

## Solución aplicada

**Tipos (commit `c110bcb`) — sin cambios de comportamiento, solo tipado/normalización:**

- `tsconfig.json` raíz: `apps/web` pasa a `exclude` (ya tenía proyecto propio con
  `"jsx": "react-jsx"` en `apps/web/tsconfig.json`; no hubo que crearlo).
- **`apps/api/src/params.ts` (nuevo):** helper `pathParam(req, name)` que normaliza
  `string | string[]` → `string` (Express 5 tipa así `req.params` por los comodines `*splat`;
  los params con nombre del contrato son siempre string). Aplicado en los 23 sitios de
  bots/battles/catalog/standings.
- `maps.ts`: `map.meta?.supportedModes ?? []` — el campo vive en `meta` según `InternalMap`
  (así lo escriben el generador de E4 y el importador de Tiled, y así lo lee el validador);
  leerlo de la raíz siempre daba `undefined` y persistía `[]`. *Es el único fix que corrige
  además un bug latente de datos (antes `supported_modes` se guardaba siempre vacío).* Ningún
  test dependía del valor vacío.
- `battle.ts` (⚠️ **solape con R2.7**, ver Notas): `new Set(...)` → `new Set<string>(...)`.
  `cmd` es `any`, así que `.filter` devuelve `any` y el constructor infería `Set<unknown>`.
  Cambio de **una anotación de tipo + un comentario**; cero lógica.
- `supervisor.ts`: `SpawnedProcess.kill(signal?: NodeJS.Signals)` — es lo que acepta el
  `ChildProcess` real; los llamantes ya pasaban `"SIGTERM"` literal y el mock del test sigue
  siendo compatible (parámetro más ancho).
- `redis-signal.ts`: `(chunk: Buffer)` en el listener de `data` (sin encoding configurado el
  socket siempre entrega `Buffer`).
- `battle-stats-canonical.test.ts`: el resolver pasa `botId` al agente (`BotAgent` lo exige).

**CI (commit `5781faa`):**

- Paso de tipos **sin `continue-on-error`** y ejecutando ambos proyectos:
  `npm run typecheck` = `tsc --noEmit && tsc --noEmit -p apps/web`. Cierra H7/issue #11.
- Paso "Formato" real: `npm run format:check` (= `prettier --check .`) con `prettier@3.9.5`
  como devDependency explícita y config del repo (`.prettierrc.json`, `printWidth: 120`,
  resto por defecto — coincide con el estilo dominante: comillas dobles, punto y coma).
- Ámbito del formateador en `.prettierignore`: **solo código TS/JS**. Fuera JSON/MD/YAML (los
  JSON de mapas/fixtures participan en checksums; la prosa de docs no debe churnar),
  `apps/arena-engine/` (temporal, R2.7 en curso) y el prototipo v1.
- Cobertura: las unitarias corren con `--coverage.*` (proveedor V8, `@vitest/coverage-v8` como
  devDependency) y publican `coverage/` como artefacto `coverage-report` (`upload-artifact@v4`,
  `if: always()`). `coverage/` añadido a `.gitignore`.

**Formato (commit `17a2849`):** `prettier --write .` mecánico, separado para no ensuciar el diff:
137 ficheros, sin cambios de lógica. Dos ficheros (`public-api.test.ts`, `replay-service/src/server.ts`)
necesitaron doble pasada (no-idempotencia conocida de prettier); tras ella `--check` da limpio.

### Demostración del DoD "la CI falla si se introduce un error de tipos"

Ejecutada en local con el comando exacto del workflow:

1. Se añadió a `apps/api/src/params.ts` la línea `const _r21Demo: number = "no soy un number";`
2. `npm run typecheck` → `error TS2322: Type 'string' is not assignable to type 'number'` →
   **exit 1** (sin `continue-on-error`, ese exit rompe el job `lint-format-types`).
3. Se retiró la línea → `npm run typecheck` exit 0.

Análogo para formato: antes del commit mecánico, `npm run format:check` salía con exit 1
listando los ficheros sin formatear; tras formatear, exit 0.
