# Reporte R2.3 — Robustez de la suite en Windows: separación por dependencia de BD

- **Tarea:** R2.3 (Ronda 2) · **Error:** ERR-GES-04 · **Equipo:** E7/E12
- **Severidad:** Media (gestión/entorno) · **Estado:** ✅ Hecho y verificado (Linux, ambos modos)
- **Fecha:** 2026-07-17 · **Rama:** `ronda2/r2.3-split-suite` (base `ronda2/entrypoints-servicios`)

## Problema

`npm test` mezclaba tests puros con tests que arrancan PostgreSQL embebido
(`embedded-postgres`). En Windows `pg_ctl` embebido no funciona, así que **toda** la
suite quedaba inservible: no se podía ejecutar ni la parte del motor/validadores/SDK,
que no necesita BD. No es un problema de lógica de tests sino de entorno y organización
de la suite (ERR-GES-04).

## Recuento real

El dosier hablaba de 22 ficheros y una pasada previa contó 18. El número real en esta
base es **20 ficheros de test** que usan `startTestDb` (único punto de arranque de BD
embebida del repo, `apps/api/src/testing/test-db.ts`):

8 en `apps/api`, 6 en `apps/tournament-worker`, 1 en `apps/replay-service`,
1 en `apps/web`, y 4 en `tests/` (acceptance, e2e ×2, gamedays).

## Solución aplicada

**Sin tocar la lógica de ningún test.** Cuatro piezas:

1. **Etiquetado automático** — `scripts/list-db-tests.mjs` escanea los ficheros de
   test y marca como "de BD" los que contienen `startTestDb`: exactamente el criterio
   por el que fallan en Windows. No hay lista manual que mantener; un test nuevo con
   `startTestDb` cambia de suite solo.
2. **Suites separadas** — `vitest.pure.config.ts` (base menos los 20 de BD) y
   `vitest.db.config.ts` (solo los 20). Scripts npm: `test:pure` y `test:db`;
   `npm test` sigue ejecutando todo, sin cambios. La suite de BD **falla cerrado**:
   si el escaneo devolviera 0 ficheros, la config lanza en vez de dar verde vacío.
3. **Fallback `DATABASE_URL`** — si está definida, `startTestDb` no arranca el
   embebido: se conecta a ese servidor, crea una BD efímera de nombre único por test
   (`arena_test_<16 hex>`, aislamiento entre ficheros en paralelo), migra, y en
   `stop()` la borra con `DROP DATABASE … WITH (FORCE)`. Requiere `CREATEDB`.
   Respeta `opts.migrate`. Sin `DATABASE_URL`, el camino embebido queda idéntico.
4. **Documentación** — `docs/getting-started.md` (nuevo): requisitos, las dos
   mitades de la suite, camino embebido (Linux/macOS), camino `DATABASE_URL` en
   Windows (contenedor o servicio local) y el motivo; y qué reportar cuando no hay
   PostgreSQL disponible (omitido, nunca aprobado).

## Verificación (Linux, Node 20.19.2, sin Docker ni sudo)

Línea base en la rama base, antes de tocar nada: **706 pasan · 1 falla · 3 skipped**
(76 ficheros). El fallo es el conocido y aceptado de `zstd` en `replay-golden.test.ts`
(necesita Node ≥ 22.15).

| Comando | Modo | Resultado |
|---|---|---|
| `npm run test:pure` | sin BD | 56 ficheros · **578 pasan · 1 falla (zstd conocido) · 3 skipped** |
| `npm run test:db` | embedded-postgres | 20 ficheros · **128 pasan · 0 fallan** |
| `npm run test:db` | `DATABASE_URL` → PostgreSQL externo real (18.4 standalone en 127.0.0.1:54329) | 20 ficheros · **128 pasan · 0 fallan**; tras la suite, **0 bases `arena_test_*` huérfanas** en el servidor (limpieza verificada) |
| `npm test` | todo | **706 pasan · 1 falla (zstd conocido) · 3 skipped** — idéntico a la base, delta 0 |

578 + 128 = 706: la partición es exhaustiva y sin solapes. **Ambos modos de BD
verificados en vivo** en esta máquina; lo único no probado aquí es Windows en sí
(no hay máquina Windows en este entorno), que es justo el escenario al que sirve el
camino `DATABASE_URL` documentado.

## Límites conocidos

- El marcador de detección es textual (`startTestDb` en el fichero). Es el mismo
  criterio del fallo real y falla cerrado en vacío, pero un helper intermedio que
  envolviera `startTestDb` con otro nombre no se detectaría; hoy no existe ninguno.
- Con `DATABASE_URL` definida, el proceso de test entero la ve: código que llamara a
  `createDb()` sin argumento se conectaría a la BD de administración en vez de fallar.
  Los tests actuales pasan siempre la URL del handle, así que no aplica.
- Windows real no verificado en esta máquina (no disponible); el mecanismo que usa
  ese camino sí, contra un PostgreSQL externo real.
