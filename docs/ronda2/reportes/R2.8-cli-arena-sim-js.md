# R2.8 · Paridad de SDK: CLI `arena-sim` en JavaScript (deuda de E5)

**Estado:** ✅ Hecho y verificado (Linux, Node 20, sin Docker).

**Origen:** carencia declarada honestamente en `docs/sdk-paridad.md` (fila "CLI de un
solo bot") y en "Lo que queda fuera" de `docs/historial/entrega-E5.md`: el SDK de
Python instala `arena-sim` como entry point de pip; el de JS tenía toda la lógica en
`startLocalBattle` (tests) pero nadie había escrito el envoltorio de CLI.

## Qué se ha hecho

### 1. La lógica compartida sube de los tests al SDK (`src/local-simulator.ts`)

`startLocalBattle` vivía en `sdks/javascript/tests/helpers.ts`. Un CLI no debe
importar de `tests/`, así que la implementación se **movió** a
`sdks/javascript/src/local-simulator.ts` (reutilizar, no reimplementar — el mismo
`Battle` + `ProtocolServer` reales, en proceso) y se generalizó con lo que el CLI de
Python ya ofrecía y el helper no:

- mapas `empty|mvp|ctf` (antes solo `empty`),
- stubs `idle|hunter|circle|forward` (antes solo `idle|hunter`),
- `ruleset` configurable (antes `dm_practice@1` fijo),
- ritmo configurable: sin `tickIntervalMs` corre al ritmo real del motor (~33 ms/tick).

`tests/helpers.ts` queda como envoltorio fino que inyecta los defaults rápidos de
test (3 ms/tick, deadline 60 ms) que los tests existentes ya asumían — cero cambios
de comportamiento en `contract.test.ts`, `gunner.test.ts` y `miner.test.ts`.

### 2. El CLI (`src/arena-sim.ts`)

`sdks/javascript/src/arena-sim.ts`, registrado como `bin` del workspace: tras
`npm install`, `npx arena-sim …` funciona desde la raíz del repo (paralelo honesto
del entry point de pip, dentro de lo que permite un paquete `private`).

```bash
npx arena-sim example-bots/javascript/gunner.ts --archetype gunner --opponent idle
```

Misma interfaz conceptual que `sdks/python/arena_sdk/simulator.py::cli`:

- carga el módulo del usuario y usa la **primera subclase de `ArenaBot`** exportada
  (mismo criterio que el `issubclass` de Python; con red de seguridad por
  duck-typing por si el bot carga otra copia del SDK y `instanceof` cruza clases),
- mismos flags: `--archetype`, `--opponent idle|hunter|circle|forward`,
  `--opponent-archetype`, `--map`, `--ruleset`, `--ticks`, `--seed`,
- misma salida: el `BattleResult` en JSON por stdout,
- mismos botIds (`bot_cli01` / `bot_opp01`, válidos contra `hello.schema.json`).

Extra solo de JS, documentado como diferencia: `--tick-interval-ms` (con deadline
`max(80, 6×intervalo)`, la misma regla que `local-sim.ts` usa para los tests).

### 3. Test E2E (`tests/arena-sim-cli.test.ts`)

Cruza de verdad el límite de proceso: `spawnSync` de `tsx src/arena-sim.ts` con el
**gunner de ejemplo** (`example-bots/javascript/gunner.ts`), como lo usará un
usuario. Asserts: exit 0, `BattleResult` parseable, `versions.protocol === "arena/1"`,
`ticks` dentro del límite, `veh_1` **no descalificado** (prueba de que el bot conectó
por WebSocket y respondió a tiempo) y ganador en `{red, draw}` (contra un stub que no
dispara, `blue` delataría un bot roto; no se exige `red` a secas para no hacer flaky
el test — el DoD de winrate del gunner es estadístico, 20 batallas, y ya existe en
`gunner.test.ts` bajo `RUN_SLOW`). Más dos casos de error: módulo sin subclase de
`ArenaBot` → exit 1 con mensaje claro; sin argumentos → uso por stderr y exit 1.

En una ejecución manual con seed fija el gunner ganó por eliminación en el tick 490
de 900 — la batalla completa de verdad, no un smoke de arranque.

### 4. Documentación

- `docs/sdk-paridad.md`: la fila del CLI pasa de carencia a **cubierto**, con las
  tres diferencias honestas que quedan: (1) en proceso vs subproceso (coherente con
  la fila "Simulador local"), (2) `--tick-interval-ms` solo existe en JS, (3) pip
  instala en cualquier entorno y el bin de JS solo existe dentro del monorepo.
- `docs/historial/entrega-E5.md`: nota de cierre apuntando a R2.8 (sin reescribir
  la historia).
- `sdks/javascript/README.md`: sección de uso del CLI.

## Hallazgo colateral: el SDK estricto destapó 4 errores de tipos del motor

`sdks/javascript/tsconfig.json` compila con `strict: true`; al importar ahora el
motor desde `src/`, `tsc` lo revisó en estricto por primera vez y salieron 4 errores
reales (el root compila con `strict: false` y nunca los vio):

| Archivo | Error | Arreglo |
|---|---|---|
| `apps/arena-engine/src/protocol-server.ts` (×2) | `expectedForTick` es `number \| null` y se restaba sin narrowing | Guard explícito (`windowOpen` solo se abre tras asignarle un número; comportamiento idéntico) |
| `apps/arena-engine/src/sim/battle.ts` (×2) | `cmd.fire.filter` con `any` producía `Set<unknown>` → `slots: unknown[]` | Type guard `(s): s is string` + `Set<string>`; misma semántica |

Sin cambios de comportamiento en runtime; `npx tsc --noEmit -p tsconfig.json` del
root sigue en verde.
