# Reporte R2.2 — Semáforo verde/amarillo/rojo en la CI

- **Tarea:** R2.2 (banda R-P1) · **Error:** ERR-GES-05 (resto: la parte de tipos/formato/cobertura la cerró R2.1)
- **Severidad:** Media (gestión/CI) · **Equipo:** transversal
- **Commits:** ver PR · **Estado:** ✅ Hecho (lógica verificada con tests; el run real de Actions queda pendiente del PR)
- **Fecha:** 2026-07-17 · **Rama:** `ronda2/r2.2-semaforo-ci` (base `ronda2/r2.1-tipos-ci`, apilada sobre el PR #22)

## Problema

La CI presentaba trabajo **omitido** como **aprobado**:

1. `deploy-staging` y `smoke-and-promote`: sin `STAGING_HOST` hacían `::warning` + `exit 0` →
   el run entero salía **verde** sin haber desplegado ni pasado el humo. Un operador que mirase
   el check de main creía que había staging validado cuando no se había ejecutado nada.
2. El escaneo del Compose llevaba `if: hashFiles('…scan-compose.mjs') != ''`: si el escáner
   desapareciera (renombrado, borrado en un merge), el paso se saltaba **en silencio** y el job
   de seguridad quedaba verde.
3. Los tests de `infrastructure/` (escáner de Compose incluido) **no corrían en ninguna etapa
   de ci.yml** (unit ejecutaba solo `apps packages`).
4. No existía un veredicto único del run: "verde" mezclaba "todo probado" con "no se pudo probar".

## Solución aplicada

### Gate testeable: `infrastructure/scripts/ci-gate.mjs`

Toda la lógica del semáforo vive en un script puro (sin dependencias, exportable) que el job
final invoca con `NEEDS_JSON='${{ toJSON(needs) }}'`:

- **🟢 VERDE** — todo lo obligatorio ejecutado y aprobado; en main, además, `deploy-staging`
  y `smoke-and-promote` con **evidencia declarada** (`outputs.resultado` = `desplegado` /
  `promocionado`).
- **🟡 AMARILLO** — pruebas bien pero entorno externo no disponible: staging omitido
  (`resultado=omitido`), job de despliegue `skipped` en main, o éxito **sin** output de
  evidencia (fail-closed: un verde sin evidencia se degrada a omitido).
- **🔴 ROJO** — fallo funcional o de seguridad (`scan` en rojo se etiqueta explícitamente como
  fallo de seguridad), un job **obligatorio** `skipped`/`cancelled`, un job esperado ausente
  del contexto `needs`, un resultado desconocido o un `NEEDS_JSON` inválido. Regla de oro:
  **todo camino no verificable falla cerrado**.

En PR, `deploy-staging`/`smoke-and-promote` en `skipped` cuentan como "no aplica" (solo corren
en push a main); el semáforo de un PR cubre las etapas 1–6 + E2E.

### Cambios en `.github/workflows/ci.yml`

- **`deploy-staging`**: declara `outputs.resultado`. Sin `STAGING_HOST` → `resultado=omitido`
  (+ warning); con host y despliegue → `resultado=desplegado`. Ya no hay `exit 0` mudo.
- **`smoke-and-promote`**: ídem (`omitido`/`promocionado`) y además **no promociona sobre un
  despliegue que no ocurrió**: si `needs.deploy-staging.outputs.resultado != 'desplegado'`,
  el humo queda omitido aunque `STAGING_HOST` existiera.
- **`scan`**: fuera el `if: hashFiles(...)` — sin escáner, el job **falla** (rojo). Trivy ya
  bloqueaba (`exit-code: "1"`, CRITICAL,HIGH) y `npm audit --audit-level=high` también.
- **`unit`**: ahora ejecuta `apps packages infrastructure` (los tests del escáner, del gate,
  de backup y de observabilidad entran en la etapa 2).
- **Nuevo job `semaforo`** (`if: always()`, needs de los 9 jobs): ejecuta el gate; rojo →
  exit 1 (falla y bloquea), amarillo/verde → exit 0 con summary y anotaciones.
- **Tipos y formato**: ya eran bloqueantes en la base (R2.1, líneas del job
  `lint-format-types`); **verificado, no duplicado**.

### Cómo se materializa el "amarillo" en GitHub Actions (decisión documentada)

Actions **no permite** que un job de workflow termine con conclusión `neutral` (el exit code 78
de Actions v1 ya no existe). El mecanismo más honesto disponible, y el elegido:

1. **Check-run "Semáforo CI"** creado vía API (`actions/github-script`, permiso
   `checks: write`) con conclusión `success` / **`neutral`** / `failure` según el output
   `color` del gate. El neutral es el amarillo real de la UI de GitHub (gris/neutro, no verde).
2. **Step summary** del job `semaforo` con la tabla job-a-job (resultado, color, motivo).
3. **Anotaciones** `::warning::` por cada omisión y `::error::` por cada rojo.
4. El **rojo hace fallar el job** `semaforo` (bloqueante como required check).

El paso que publica el check-run lleva `continue-on-error: true`: la fuente de verdad es el
propio job (exit/summary/warnings); así un token de solo lectura (PR de fork) no convierte un
veredicto válido en fallo de fontanería. Alternativas descartadas: fallar el job en amarillo
(convertiría "entorno pendiente" en bloqueo permanente de main, tan deshonesto como el verde
falso) y `continue-on-error` a nivel de job (pinta el run de verde con una nota minúscula).

## Omitido y por qué

- **Ejecución real en Actions**: esta VM no puede correr los workflows (sin Docker, sin
  runners). El comportamiento del gate está probado invocando el CLI exactamente como lo hace
  el workflow (mismo contrato `NEEDS_JSON`/`GITHUB_OUTPUT`/`GITHUB_STEP_SUMMARY`); el YAML
  está parseado pero no linted (`actionlint` no disponible). El primer push del PR es la
  verificación viva de las etapas 1–6 + semáforo; el camino amarillo de main solo se verá
  tras el merge (staging sigue sin configurar → debe salir 🟡, no 🟢).
- **`e6-security.yml` / `nightly.yml` / `acceptance.yml`**: sin cambios. E6 ya aplica su
  propio fail-closed (digests-gate de R1.6/R6.1, jobs `skipped` explícitos) y no presenta
  omisiones como éxito; unificarlos bajo el mismo gate sería alcance nuevo, no ERR-GES-05.
- El check-run "Semáforo CI" no puede marcarse **required** desde el repo (es configuración
  de branch protection del operador); se recomienda exigir `semaforo` como required check.

## Seguimiento

- Al configurar `STAGING_HOST`/`STAGING_SSH_KEY`, descomentar el bloque ssh del deploy: el
  semáforo pasará de 🟡 a 🟢 solo con evidencia real.
- Si se añade o renombra un job en `ci.yml`, actualizar `JOBS_CI` en `ci-gate.mjs` (un job
  esperado ausente sale en rojo, así que la deriva se detecta sola en la primera ejecución).
